# Ответы на вопросы по проекту: Система учёта инцидентов безопасности

---

## 1. Какую задачу (основная функциональность) решает ваше приложение?

Приложение представляет собой систему учёта инцидентов безопасности в промышленности и на производстве. Основная задача — предоставить пользователю удобный инструмент для регистрации, просмотра, редактирования и удаления записей об инцидентах: утечках химикатов, превышениях уровня шума, взрывах, нарушениях техники безопасности и т.п.

Приложение позволяет отслеживать статистику инцидентов по уровню опасности (критический, высокий, средний, низкий), видеть детальную информацию по каждому случаю, а также фиксировать принятые меры по устранению последствий. На главной странице отображается сводная таблица инцидентов и статистика по категориям.

---

## 2. Какие сущности/данные обрабатывает приложение?

Приложение работает с одной основной сущностью — **Инцидент** (`incident`). Каждый инцидент хранится в JSON-базе (`db.json`) и содержит поля: `id`, `name`, `description`, `location`, `severity`, `date`, `measures`. Все операции (создание, чтение, обновление, удаление) выполняются именно над этой сущностью.

Дополнительных сущностей (пользователи, категории и т.п.) в текущей версии нет — приложение сфокусировано на одной предметной области. Данные хранятся и обрабатываются на стороне простого REST API-сервера (`json-server`), а клиент работает с ними через HTTP-запросы.

## 3. Какие основные маршруты (роуты) есть в вашем SPA и почему вы выбрали именно их?

В приложении определены следующие маршруты (см. `App.js`):
- `/` — Главная страница со статистикой и кратким списком инцидентов;
- `/detail` — Список инцидентов для просмотра; `/detail/:id` — детальная карточка конкретного инцидента;
- `/edit` — Список инцидентов для редактирования; `/edit/:id` — форма редактирования конкретного инцидента;
- `/delete` — Список инцидентов для удаления; `/delete/:id` — страница подтверждения удаления;
- `/add` — Форма добавления нового инцидента.

Маршруты выбраны исходя из основных операций CRUD (Create, Read, Update, Delete). Разделение на «список» и «конкретный элемент» (с `:id`) позволяет реализовать двухуровневую навигацию: сначала пользователь выбирает нужный инцидент из списка, затем переходит к нужному действию. Такой подход делает интерфейс интуитивно понятным и предотвращает случайные операции над данными.

## 4. Как реализован роутинг: используете ли вы hash-роутинг или History API?

В приложении используется **History API** через компонент `BrowserRouter` из библиотеки `react-router-dom` (строка `import { BrowserRouter as Router, ... } from 'react-router-dom'` в `App.js`). Это означает, что URL-адреса выглядят как обычные пути (например, `/detail/1`), без символа `#`.

History API выбран как более современный и «чистый» подход к маршрутизации: URL выглядят профессионально и привычно для пользователя. В отличие от hash-роутинга (`/#/detail/1`), такие ссылки лучше воспринимаются людьми и корректно индексируются поисковыми системами. При необходимости деплоя на сервер нужно лишь настроить перенаправление всех запросов на `index.html`.

## 5. Какие HTTP-методы вы используете (GET, POST, PUT, DELETE) и для чего каждый из них?

В файле `IncidentsContext.js` используются четыре HTTP-метода:
- **GET** — `axios.get(API_URL)` для получения всего списка инцидентов (`fetchIncidents`) и `axios.get(\`${API_URL}/${id}\`)` для получения одного инцидента по ID (`fetchIncidentById`);
- **POST** — `axios.post(API_URL, ...)` для создания нового инцидента (`addIncident`);
- **PUT** — `axios.put(\`${API_URL}/${id}\`, ...)` для полного обновления данных инцидента (`updateIncident`);
- **DELETE** — `axios.delete(\`${API_URL}/${id}\`)` для удаления инцидента (`deleteIncident`).

Выбор методов соответствует принципам REST: каждый метод имеет семантически чёткую роль. GET идемпотентен и используется только для чтения. POST создаёт новый ресурс. PUT полностью заменяет существующий ресурс. DELETE удаляет ресурс. Такой подход облегчает понимание кода и делает API предсказуемым.

## 6. Почему в качестве формата обмена данными между клиентом и сервером выбран JSON?

JSON (JavaScript Object Notation) выбран как формат обмена данными, потому что он является стандартом де-факто для REST API и нативно поддерживается JavaScript. В коде при отправке POST и PUT запросов явно указывается заголовок `"Content-Type": "application/json"` и используется `JSON.stringify()` для сериализации объектов (см. `IncidentsContext.js`).

JSON удобочитаем для разработчика, легко отлаживается в браузерных инструментах разработчика, а `json-server` (используемый как бэкенд) работает именно с JSON. Axios автоматически парсит JSON-ответы сервера, что избавляет от необходимости ручного вызова `.json()`. В совокупности это делает JSON оптимальным выбором для данного проекта.

## 7. Как обрабатываются возможные ошибки или неуспешные ответы?

Обработка ошибок централизована в функции `handleApiError` внутри `IncidentsContext.js`. Функция анализирует объект ошибки axios: если есть `err.response`, извлекается HTTP-статус и возвращается соответствующее русскоязычное сообщение для кодов 400, 403, 404, 500, 502, 503. Если ответа нет (`err.request`), сообщается о проблемах с сетью. В остальных случаях — общая ошибка запроса.

На уровне компонентов ошибки отображаются через компонент `ErrorMessage`, который показывает текст ошибки и кнопки «Повторить» и «Закрыть». Для inline-ошибок (например, при загрузке данных в фоне) используется `error-banner` — полоса с предупреждением поверх контента. Состояние `error` хранится в контексте и очищается вызовом `clearError()`.

## 8. Каким образом вы передаете и получаете данные: используете fetch, axios или иной инструмент?

В приложении используется библиотека **axios** (зависимость `"axios": "^1.13.6"` в `package.json`). Все HTTP-запросы к API инкапсулированы в функциях контекста (`fetchIncidents`, `fetchIncidentById`, `addIncident`, `updateIncident`, `deleteIncident`) в файле `IncidentsContext.js`.

Axios выбран вместо нативного `fetch` по ряду причин: он автоматически парсит JSON-ответы, имеет более удобный API для передачи заголовков и тела запроса, а также лучшую обработку ошибок (axios выбрасывает исключение при статусах 4xx/5xx, тогда как `fetch` этого не делает). Кроме того, axios поддерживает перехватчики (interceptors), что упрощает будущее добавление авторизации или глобального логирования запросов.

## 9. Как организовано хранение состояния (state) в вашем приложении?

Состояние приложения хранится в **React Context** (`IncidentsContext.js`). Провайдер `IncidentsProvider` оборачивает всё приложение в `App.js` и предоставляет дочерним компонентам общее состояние через хук `useIncidents()`. В контексте хранятся: массив инцидентов `incidents`, текущий выбранный инцидент `currentIncident`, флаг загрузки `loading` и строка ошибки `error`.

Локальное состояние компонентов (`useState`) используется для: данных форм (`formData`, `validationErrors`), флагов отправки (`isSubmitting`, `isDeleting`), локальных ошибок загрузки (`loadError`) и флага `isLoaded` в `Edit.js`. Такое разделение позволяет хранить разделяемое состояние глобально, а временное — локально, не засоряя контекст.

## 10. Используете ли вы глобальный менеджер состояния (Redux, Vuex, MobX, Pinia и т. п.) или локальное состояние?

В приложении **не используется** сторонний глобальный менеджер состояния (Redux, MobX и т.п.). Вместо этого применяется встроенный механизм React — **Context API** в сочетании с хуками `useState` и `useCallback`. Это вполне достаточно для приложения с одной сущностью и небольшим набором операций.

Контекст (`IncidentsContext`) выполняет роль «глобального хранилища» — все страницы получают доступ к инцидентам и функциям API без пробрасывания пропсов через промежуточные компоненты. Хук `useCallback` используется для мемоизации функций-запросов, что предотвращает лишние ре-рендеры. При росте приложения контекст можно заменить на Redux Toolkit без изменения интерфейса хука `useIncidents`.

## 11. Какие поля (атрибуты) вы храните у ваших сущностей и почему именно они необходимы?

Каждый инцидент содержит следующие поля:
- `id` — уникальный идентификатор, необходим для адресации ресурса в REST API;
- `name` — краткое название инцидента для отображения в списках;
- `description` — подробное описание произошедшего;
- `location` — место происшествия (цех, участок, склад);
- `severity` — уровень опасности (`низкий`, `средний`, `высокий`, `критический`);
- `date` — дата инцидента для хронологической сортировки и анализа;
- `measures` — принятые меры (необязательное поле, может быть пустым).

Все поля отражают реальные требования к документированию производственных инцидентов: где произошло, что произошло, насколько опасно, когда и что было предпринято. Поле `severity` позволяет строить статистику и визуально выделять опасные инциденты цветом. Поле `measures` необязательно, так как меры могут быть ещё не приняты на момент регистрации.

## 12. Применяете ли вы валидацию данных на клиенте при добавлении/редактировании сущностей?

Да, клиентская валидация реализована в функциях `validateForm()` в компонентах `Form.js` и `Edit.js`. Проверяются следующие условия: `name` — обязательное, минимум 3 символа, максимум 100; `description` — обязательное, минимум 10 символов; `location` — обязательное, минимум 3 символа; `date` — обязательная, не может быть в будущем; `severity` — должно совпадать с одним из допустимых значений.

При нарушении любого правила поле подсвечивается красной рамкой (класс `input-error`), под полем появляется текст ошибки (компонент с классом `validation-error`), а форма не отправляется на сервер. При исправлении поля ошибка сразу исчезает благодаря обработчику `handleChange`, который очищает ошибку валидации при изменении соответствующего поля.

## 13. Как устроена форма создания/редактирования сущности (какие поля ввода, проверка и т. д.)?

Форма (компонент `Form.js` для создания, `Edit.js` для редактирования) содержит следующие поля: текстовый `input` для названия, `textarea` для описания (4 строки), `input` для места происшествия, `select` для уровня опасности с 4 вариантами (низкий/средний/высокий/критический), `input[type="date"]` с ограничением `max` на текущую дату, и `textarea` для принятых мер (3 строки, необязательное).

Каждое обязательное поле помечено звёздочкой `*`. Все поля управляются через единый обработчик `handleChange`, который обновляет объект `formData` в `useState`. При редактировании форма предзаполняется данными инцидента, загруженными через `fetchIncidentById`. Форма имеет атрибут `noValidate` для отключения браузерной валидации — всю проверку берёт на себя кастомная `validateForm()`.

## 14. Каким образом вы обрабатываете отправку формы на сервер (функция-обработчик, события, хуки и т. д.)?

Отправка формы обрабатывается асинхронной функцией `handleSubmit`, которая назначена обработчиком события `onSubmit` тега `<form>`. Первым делом вызывается `e.preventDefault()` для отмены стандартного поведения браузера. Затем выполняется `validateForm()` — если есть ошибки, функция возвращает `false` и отправка отменяется.

При успешной валидации устанавливается флаг `isSubmitting = true` (отображается спиннер), после чего вызывается `addIncident` или `updateIncident` из контекста — эти функции выполняют реальный HTTP-запрос через axios. В случае успеха показывается `alert` с подтверждением и происходит навигация через `useNavigate()`. В блоке `finally` флаг `isSubmitting` сбрасывается независимо от результата.

## 15. Что происходит в приложении после успешного добавления/редактирования/удаления сущности? Как обновляется UI?

После успешного выполнения операции на сервере контекст обновляет локальный массив `incidents`:
- При добавлении: `setIncidents(prev => [...prev, response.data])` — новый инцидент добавляется в конец массива;
- При редактировании: `setIncidents(prev => prev.map(item => item.id === numId ? response.data : item))` — заменяется один элемент;
- При удалении: `setIncidents(prev => prev.filter(item => item.id !== numId))` — элемент исключается из массива.

После обновления состояния React автоматически перерендерит все компоненты, использующие данные из контекста. Пользователю показывается `alert` с сообщением об успехе, а затем через `useNavigate()` происходит перенаправление на список (например, `/edit`, `/delete` или `/`). Таким образом список автоматически отображает актуальное состояние без необходимости делать повторный запрос к серверу.

## 16. Как вы организовали структуру файлов и директорий в проекте?

Структура проекта (`src/`) разделена на логические директории:
```
src/
├── components/     # Переиспользуемые UI-компоненты
│   ├── Spinner.js / Spinner.css   # Индикатор загрузки
│   └── ErrorMessage.js / ErrorMessage.css  # Отображение ошибок
├── context/        # Глобальное состояние и API-слой
│   └── IncidentsContext.js
```

Такое разделение следует распространённым React-конвенциям: `components` — для мелких переиспользуемых элементов, `pages` — для страниц, соответствующих маршрутам, `context` — для управления состоянием. Это упрощает навигацию по проекту и масштабирование: новые страницы добавляются в `pages/`, новые общие компоненты — в `components/`.

## 20. Какие трудности возникли при интеграции фронтенда и бэкенда (RESTful API) и как вы их решили?

Одной из основных трудностей стала работа с `id`: `json-server` может возвращать идентификаторы как числа, а компонент `useParams()` из react-router-dom возвращает их как строки. Это приводило к тому, что `item.id === numId` не срабатывало при сравнении. Решение — явное приведение типов: `const numId = typeof id === 'string' ? parseInt(id) : id` в функциях `updateIncident` и `deleteIncident`.

Другой трудностью стало управление `id` при создании нового инцидента: `json-server` должен генерировать `id` автоматически, однако в коде вручную вычисляется `maxId` из текущих инцидентов и увеличивается на 1. Это позволяет контролировать значение `id` на клиенте, но требует, чтобы список инцидентов был актуален до отправки POST-запроса. Для полного устранения этой проблемы стоит доверить генерацию `id` серверу и убрать эту логику с клиента.

## 21. Используете ли вы для работы с API собственный «слой» (класс или модуль для запросов) или вызываете fetch/axios напрямую из компонентов?

В приложении реализован **API-слой через React Context** — все вызовы axios сосредоточены исключительно в `IncidentsContext.js`. Компоненты (страницы) никогда не обращаются к axios напрямую: они вызывают только функции из контекста (`fetchIncidents`, `addIncident`, `updateIncident`, `deleteIncident`). Это чёткое разделение ответственности между «слоем данных» и «слоем представления».

Такой подход имеет несколько преимуществ: единственное место для изменения базового URL (`BASE_URL`), единый обработчик ошибок (`handleApiError`), возможность легко заменить axios на fetch или другую библиотеку без изменения компонентов. При масштабировании этот «слой» можно вынести в отдельный файл `api/incidentsApi.js` или превратить в класс, а контекст оставить только для хранения состояния.

## 22. Как вы планируете масштабировать приложение, если понадобится добавить новые сущности или функциональность?

Текущая архитектура хорошо масштабируется горизонтально. Для добавления новой сущности (например, «Плановые проверки» или «Сотрудники»): создаётся новый контекст по образцу `IncidentsContext.js`, добавляются новые страницы в `src/pages/`, регистрируются новые маршруты в `App.js`. Благодаря единой структуре все новые сущности будут иметь предсказуемое поведение.

При значительном росте сложности рекомендуется: заменить Context API на Redux Toolkit (легко, так как интерфейс `useIncidents` можно сохранить), вынести API-функции в отдельный сервисный слой `src/services/`, добавить TypeScript для типобезопасности данных. Базовый URL уже вынесен в переменную окружения `REACT_APP_API_URL`, что позволяет без изменения кода переключаться между средами (dev/staging/production).

## 23. Есть ли у вас обработка нестандартных статус-кодов сервера (например, 401 Unauthorized) и как она реализована?

Обработка HTTP-статусов реализована в функции `handleApiError` в `IncidentsContext.js`. Функция обрабатывает статусы: 400 (некорректный запрос), 403 (доступ запрещён), 404 (ресурс не найден), 500 (внутренняя ошибка сервера), 502 и 503 (сервер недоступен). Статус **401 Unauthorized** явно не обрабатывается, так как приложение не имеет системы авторизации, и `json-server` по умолчанию не возвращает 401.

Для любого необработанного кода срабатывает ветка `default`, которая возвращает сообщение `Ошибка сервера: ${status}`. При отсутствии ответа от сервера (`err.request`) пользователь видит сообщение о проблемах с сетью. При необходимости добавить авторизацию достаточно: добавить обработку 401 в `handleApiError` с перенаправлением на страницу логина, и настроить axios-интерцептор для автоматической вставки JWT-токена в заголовки запросов.
